## 1. Что мы разворачиваем (простыми словами)

Ваш проект — **два отдельные «сервисы»** + **база данных**:

```
┌─────────────────────────────────────────────────────────────┐
│  Пользователь открывает в браузере                          │
│  https://myfirstproject.su  (или *.timeweb.cloud)           │
│                    │                                        │
│                    ▼                                        │
│  ┌──────────────────────────────────────┐                 │
│  │  ФРОНТЕНД (React)                     │  Timeweb Apps   │
│  │  Кнопки, формы, страницы             │  папка frontend │
│  │  Статические файлы после сборки      │                 │
│  └──────────────────────────────────────┘                 │
│                    │                                        │
│                    │ запросы API (fetch)                    │
│                    ▼                                        │
│  ┌──────────────────────────────────────┐                 │
│  │  БЭКЕНД (FastAPI)                    │  Timeweb Apps   │
│  │  Логин, проекты, шаблоны, JWT        │  корень репо    │
│  └──────────────────────────────────────┘                 │
│                    │                                        │
│                    ▼                                        │
│  ┌──────────────────────────────────────┐                 │
│  │  PostgreSQL                          │  Timeweb БД     │
│  │  Пользователи, проекты, тарифы       │                 │
│  └──────────────────────────────────────┘                 │
└─────────────────────────────────────────────────────────────┘
```

| Часть | Что делает | Где будет жить |
|-------|------------|----------------|
| **Фронтенд** | Всё, что видит пользователь в браузере | Timeweb **Apps** (Docker, папка `frontend`) |
| **Бэкенд** | API: регистрация, сохранение проектов | Timeweb **Apps** (Docker, корень репозитория) |
| **PostgreSQL** | Хранит данные (не в файлах на диске) | Timeweb **Базы данных** |

**Важно:** фронтенд **не** обращается к базе напрямую. Он звонит бэкенду по URL API. Бэкенд читает/пишет в PostgreSQL.

**Сейчас у вас (до переезда):**
- Фронт → Vercel
- Бэкенд → Amvera (`conslanback-vladislav7599.amvera.io`)
- Домен → возможно `myfirstproject.su` на Reg.ru

**После переезда:**
- Фронт + бэкенд + БД → Timeweb

## 3. Регистрация и вход в Timeweb

1. Откройте [https://timeweb.cloud](https://timeweb.cloud).
2. Нажмите **Регистрация** или **Войти**.
3. Пополните баланс: **Баланс** → **Пополнить** (нужно для создания Apps и БД).

   - **Базы данных** — PostgreSQL

Названия пунктов могут слегка отличаться — ищите похожие.

## 4. Шаг 1 — База данных PostgreSQL

### Зачем

Бэкенд хранит пользователей, проекты, тарифы в **PostgreSQL**. Без `DATABASE_URL` приложение не запустится в продакшене (или будет пустая SQLite — на Timeweb так не работаем).

### Действия в панели Timeweb

1. **Базы данных** → **Создать** → **PostgreSQL**.
2. Выберите минимальный тариф (для старта достаточно).
3. Задайте имя кластера, например `landing-constructor-db`.
4. Дождитесь создания (1–5 минут).
5. Откройте кластер → найдите **строку подключения** / **Connection string**.

Она выглядит так:

```text
postgresql://USER:PASSWORD@HOST:5432/DATABASE
```

Пример (ваши значения будут другие):

```text
postgresql://gen_user:Ab12cd34@192.168.0.5:5432/default_db
```

### Что сделать с этой строкой

1. **Скопируйте** строку целиком в блокнот.
2. **Никуда в код не вставляйте** — только в переменные окружения Timeweb (шаг 5).
3. Это секрет: не публикуйте в GitHub, не отправляйте в чаты.

### Что произойдёт при первом запуске бэкенда

Код в `backend/main.py` при старте:
- создаёт таблицы (`init_db`);
- заполняет демо-данные (`seed_db`) — планы Starter/Pro/Enterprise, 6 шаблонов, тестовые пользователи.

Если нужны **реальные данные с Amvera** — см. [шаг 6](#9-шаг-6--перенос-данных-с-amvera). Импорт делайте **до** или **вместо** первого «пустого» запуска.

## 5. Шаг 2 — Бэкенд (API) в Cloud Apps

### 5.1. Подключить GitHub к Timeweb (если ещё не подключен)

1. **Apps** → **Создать приложение**.
2. При запросе репозитория — **Подключить GitHub**.
3. Разрешите Timeweb доступ к репозиторию с проектом `secondProjectMain2` (или как он назван у вас).
4. Выберите этот репозиторий.

### 5.2. Тип приложения — Docker

1. Выберите способ деплоя: **Docker** / **Из Dockerfile**.
   - Документация Timeweb: [деплой из Dockerfile](https://timeweb.cloud/docs/apps/deploying-with-dockerfile).
2. **Не** выбирайте «голый Python FastAPI» с командой `uvicorn main:app` — в нашем проекте импорты идут как `backend.main`, и готовый Dockerfile в **корне** репозитория уже всё настроил.

### 5.3. Настройки репозитория — что вписать в каждое поле

| Поле в панели Timeweb | Что вписать | Почему |
|------------------------|-------------|--------|
| **Репозиторий** | Ваш GitHub репозиторий | Откуда Timeweb берёт код |
| **Ветка** | `main` (или `master`) | Какая ветка деплоится |
| **Путь к директории проекта** | **Пусто** (ничего не пишите) | `Dockerfile` лежит в **корне** репозитория |
| **Автодеплой** | Включить | При push в Git — автоматическая пересборка |

Файл `Dockerfile` в корне репозитория уже содержит:

- установку Python-зависимостей из `backend/requirements.txt`;
- запуск `uvicorn backend.main:app` на порту **80**.

Timeweb смотрит на `EXPOSE 80` в Dockerfile и направляет внешний трафик на этот порт.

### 5.4. Переменные окружения бэкенда

В настройках приложения найдите **Переменные окружения** / **Environment variables**.

Добавьте **каждую** строку: **Имя** (Name) | **Значение** (Value).

---

#### `DATABASE_URL` — строка подключения к PostgreSQL

| | |
|---|---|
| **Откуда взять** | Шаг 1 — панель PostgreSQL в Timeweb, «строка подключения» |
| **Куда вставить** | Backend Apps → Переменные окружения → Name: `DATABASE_URL` |
| **Пример** | `postgresql://gen_user:пароль@хост:5432/default_db` |

---

#### `JWT_SECRET` — секретный ключ для токенов входа

| | |
|---|---|
| **Что это** | Случайная длинная строка. Без неё логин небезопасен |
| **Откуда взять** | Сгенерировать на компьютере (один раз) |
| **Куда вставить** | Backend Apps → `JWT_SECRET` |

**Windows (PowerShell):**

```powershell
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

Скопируйте вывод, например: `xK9mP2vL8nQ4wR7sT1uV5yZ0aB3cD6eF`

**Если Python не установлен:** откройте [randomkeygen.com](https://randomkeygen.com) → Code Igniter Encryption Keys → скопируйте длинный ключ.

---

#### `JWT_ALGORITHM`

| | |
|---|---|
| **Значение** | `HS256` |
| **Откуда** | Фиксированное значение из `backend/.env.example` |

---

#### `JWT_EXPIRES_MINUTES`

| | |
|---|---|
| **Значение** | `120` |
| **Что это** | Время жизни токена входа в минутах (2 часа) |

---

#### `FRONTEND_URL` — для CORS (связь с фронтом)

| | |
|---|---|
| **Что это** | Адрес сайта, с которого браузер разрешён обращаться к API |
| **Сейчас (временно)** | `http://localhost:5173` |
| **После деплоя фронта** | Обновите — см. [шаг 4](#7-шаг-4--связать-фронт-и-бэкенд-cors) |

**Важно:** URL должен быть **точным**:
- с `https://`
- **без** слэша на конце  
  ✅ `https://myfirstproject.su`  
  ❌ `https://myfirstproject.su/`

Несколько адресов — через **запятую**:

```text
https://myfirstproject.su,https://landing-front-xxxxx.timeweb.cloud,http://localhost:5173
```

#### `TELEGRAM_BOT_TOKEN` и `TELEGRAM_MANAGER_CHAT_ID` (опционально)

|---|---|
| **Зачем** | Уведомления в Telegram при регистрации и смене тарифа |
| **Откуда** | [backend/docs/TELEGRAM_SETUP.md](../backend/docs/TELEGRAM_SETUP.md) |
| **Если не нужны** | Не добавляйте — приложение работает без них |

---

#### `APP_ENV` (опционально)

| | |
|---|---|
| **Значение** | `production` |
| **Что делает** | Скрывает публичную документацию `/docs` в продакшене |

### 5.5. Запуск деплоя бэкенда

1. Нажмите **Создать** / **Запустить деплой**.
2. Откройте **Логи** / **Build logs**.
3. Ждите статус **Успешно** / **Running** (3–10 минут).

**Если ошибка в логах:**
- `DATABASE_URL` — проверьте строку подключения;
- `connection refused` — PostgreSQL ещё не создан или неверный хост;
- `ModuleNotFoundError` — путь к проекту должен быть **пустой** (корневой Dockerfile).

### 5.6. Где взять URL бэкенда

1. В карточке приложения Backend → **Домены** / **URL**.
2. Timeweb даёт адрес вида:

   ```text
   https://landing-api-xxxxx.timeweb.cloud
   ```

3. **Запишите в блокнот** — это ваш **BACKEND_URL** (без `/api` на конце).

### 5.7. Проверка бэкенда

Откройте в браузере:

```text
https://ВАШ-BACKEND_URL/docs
```

или

```text
https://ВАШ-BACKEND_URL/health
```

Должна открыться страница Swagger или JSON `{"status":"ok"}`.

**Сохраните для фронта:**

```text
VITE_API_URL = https://ВАШ-BACKEND_URL/api
```

⚠️ На конце **обязательно** `/api` — так настроен код в `frontend/src/api/apiClient.ts`.

Пример:

```text
https://landing-api-xxxxx.timeweb.cloud/api
```

## 6. Шаг 3 — Фронтенд (сайт) в Cloud Apps

Фронтенд — отдельное **второе** приложение в Apps (не путать с бэкендом).

### 6.1. Создание приложения

1. **Apps** → **Создать приложение** → **Docker**.
2. **Тот же** GitHub репозиторий.
3. **Ветка:** `main`.

### 6.2. Главное отличие от бэкенда — путь к папке

| Поле | Бэкенд | Фронтенд |
|------|--------|----------|
| **Путь к директории проекта** | **пусто** | **`frontend`** |

Timeweb будет использовать `frontend/Dockerfile` в этой папке.

### 6.3. Переменная `VITE_API_URL` — самая важная для фронта

| | |
|---|---|
| **Что это** | Адрес API, в который фронт отправляет запросы |
| **Откуда взять** | Шаг 5.7 — URL бэкенда **+ `/api`** |
| **Куда вставить** | Frontend Apps → Переменные окружения |

**Пример:**

| Name | Value |
|------|-------|
| `VITE_API_URL` | `https://landing-api-xxxxx.timeweb.cloud/api` |

**Почему на этапе сборки:** Vite «вшивает» `VITE_*` в JS при `npm run build`. Если поменяете URL бэкенда — нужен **новый деплой** фронта.

### 6.4. Запуск деплоя фронта

1. **Запустить деплой**.
2. Логи: `npm ci` → `npm run build` → nginx — без ошибок.
3. Получите URL фронта, например:

   ```text
   https://landing-front-xxxxx.timeweb.cloud
   ```

Запишите как **FRONTEND_URL** для шага 4.

### 6.5. Проверка фронта

1. Откройте URL фронта в браузере.
2. Должна загрузиться главная страница Landing Constructor.
3. Откройте страницу **Тарифы** или **Регистрация**.
4. Если видите «Failed to fetch» — ещё не настроен CORS (шаг 4).

## 7. Шаг 4 — Связать фронт и бэкенд (CORS)

### Что происходит

Браузер блокирует запросы с `https://сайт-фронта` к `https://сайт-api`, если бэкенд **явно не разрешил** этот адрес. Это **CORS**.

Код бэкенда читает список разрешённых адресов из переменной **`FRONTEND_URL`** (`backend/main.py`).

### Что сделать

1. Откройте **Backend Apps** → **Переменные окружения**.
2. Измените `FRONTEND_URL` на **реальный адрес фронта**:

   ```text
   https://landing-front-xxxxx.timeweb.cloud,http://localhost:5173
   ```

   Если уже есть свой домен:

   ```text
   https://myfirstproject.su,https://landing-front-xxxxx.timeweb.cloud,http://localhost:5173
   ```

3. **Сохраните** и **перезапустите** / **Redeploy** бэкенд.

### Проверка

1. Откройте сайт фронта.
2. `F12` → вкладка **Консоль** — не должно быть красных ошибок CORS.
3. `F12` → **Сеть (Network)** → фильтр `api` — запросы к бэкенду со статусом **200**.
4. Попробуйте **Регистрация** → форма должна отправиться без «Failed to fetch».

## 8. Шаг 5 — Свой домен (например myfirstproject.su)

Необязательный шаг. Без домена сайт работает на `*.timeweb.cloud`.

### 8.1. Схема

| Что | Домен | Указывает на |
|-----|-------|--------------|
| Сайт для людей | `myfirstproject.su` или `www` | Frontend Apps |
| API (опционально) | `api.myfirstproject.su` | Backend Apps |

Можно оставить API на `*.timeweb.cloud` — тогда в `VITE_API_URL` остаётся timeweb URL.

### 8.2. В панели Timeweb — фронтенд

1. Frontend Apps → **Домены** → **Добавить домен**.
2. Введите `myfirstproject.su` (и при необходимости `www.myfirstproject.su`).
3. Timeweb покажет, **какую DNS-запись** создать (CNAME или A).

### 8.3. В панели Timeweb — бэкенд (если нужен api.)

1. Backend Apps → **Домены** → `api.myfirstproject.su`.
2. Скопируйте инструкцию DNS.

### 8.4. На Reg.ru (где куплен домен)

1. Войти в [reg.ru](https://reg.ru) → **Мои домены** → `myfirstproject.su`.
2. **Управление зоной DNS** / **DNS-серверы и зона**.
3. Добавить записи **как указал Timeweb**:

   Пример (значения **ваши**, из панели Timeweb):

   | Тип | Поддомен | Значение |
   |-----|----------|----------|
   | CNAME | `@` или `www` | что дал Timeweb для фронта |
   | CNAME | `api` | что дал Timeweb для бэка |

4. Удалите старые записи на Vercel (`cname.vercel-dns.com`), если они ещё есть.
5. Ждите 15 минут – 24 часа (обычно 1–2 часа).

### 8.5. После привязки домена — обновить переменные

1. **Backend** `FRONTEND_URL`:

   ```text
   https://myfirstproject.su,http://localhost:5173
   ```

2. **Frontend** `VITE_API_URL` (если API на поддомене):

   ```text
   https://api.myfirstproject.su/api
   ```

3. **Redeploy** фронт и бэкенд.

SSL (HTTPS) Timeweb обычно выдаёт автоматически (Let's Encrypt).

## 9. Шаг 6 — Перенос данных с Amvera

### Нужен ли перенос?

| Ситуация | Действие |
|----------|----------|
| На Amvera только тесты, пользователей не жалко | **Не переносить** — Timeweb создаст новую БД с демо-данными |
| Есть реальные пользователи и проекты | **Перенести** PostgreSQL |

### Откуда (Amvera)

1. Панель [amvera.io](https://amvera.io) → ваш проект бэкенда.
2. **Базы данных** → PostgreSQL → **Экспорт** / бэкап.
3. Или через `pg_dump` (если Amvera даёт доступ).

Текущий бэкенд Amvera: `conslanback-vladislav7599.amvera.io`.

### Куда (Timeweb)

1. Панель Timeweb → PostgreSQL → **Импорт** / восстановление из бэкапа.
2. Или `psql` с вашим `DATABASE_URL` Timeweb.

### Порядок

1. Создать PostgreSQL на Timeweb.
2. **Импортировать** данные с Amvera **до** первого запуска бэкенда с seed  
   **или** импортировать поверх после остановки бэкенда.
3. Запустить бэкенд с `DATABASE_URL` Timeweb.

Если запустить бэкенд на пустой БД — `seed_db` добавит демо-пользователи **поверх** или создаст таблицы. Для чистого переноса лучше импорт после `init_db` или полный дамп со схемой.

## 12. Словарь терминов

| Термин | Простыми словами |
|--------|------------------|
| **API / бэкенд** | Сервер, который обрабатывает логин и данные |
| **Фронтенд** | То, что открывается в браузере |
| **Dockerfile** | Инструкция для Timeweb: как собрать и запустить приложение |
| **Переменные окружения (env)** | Секреты и настройки **в панели хостинга**, не в коде |
| **CORS** | Защита браузера; бэкенд должен «узнать» адрес фронта |
| **DNS** | Настройка «домен → сервер» на Reg.ru |
| **`VITE_API_URL`** | Адрес API для фронта; **с `/api` на конце** |
| **`FRONTEND_URL`** | Адрес сайта для бэкенда; **без `/api`** |
| **`DATABASE_URL`** | Строка подключения к PostgreSQL |
| **Redeploy** | Пересборка и перезапуск приложения в Apps |

## 13. Частые ошибки

### «Failed to fetch» в браузере

1. `VITE_API_URL` на фронте — правильный? Есть `/api` на конце?
2. `FRONTEND_URL` на бэкенде — **точный** URL из адресной строки браузера?
3. После изменений — **Redeploy** обоих приложений.

### Страница `/dashboard` — 404 при прямом открытии

SPA routing: в Apps используется `nginx.timeweb.conf` — должно работать. Если Hosting + FTP — нужен `.htaccess` в `dist/` (уже в `frontend/public/`).

### Бэкенд падает при старте

Откройте **Логи** → ищите `DATABASE_URL`, `JWT_SECRET`, ошибки PostgreSQL.

### Логин не работает после переезда

Новая БД = новые пользователи. Старые аккаунты только если перенесли данные с Amvera. JWT с Amvera не переносится — пользователи просто логинятся заново.

### Сборка фронта на Timeweb без `VITE_API_URL`

Запросы идут на `/api` на том же домене, что фронт — там нет API → ошибки. **Всегда** задайте `VITE_API_URL`.

## 14. Короткий чеклист

```
□ PostgreSQL создан → DATABASE_URL в блокнот
□ Backend Apps: Docker, путь пустой, env (DATABASE_URL, JWT_*, FRONTEND_URL временно)
□ Backend URL записан → VITE_API_URL = BACKEND_URL + /api
□ Frontend Apps: Docker, путь frontend, env VITE_API_URL
□ Frontend URL записан
□ FRONTEND_URL на бэкенде обновлён → Redeploy backend
□ Сайт открывается, регистрация работает
□ (опционально) Домен на Reg.ru → DNS → обновить env → Redeploy
□ (опционально) Импорт БД с Amvera
□ Отключить Vercel и Amvera
```
